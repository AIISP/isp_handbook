## §1 Theoretical Foundation: Parameter Space and the Need for Version Management

### 1.1 Dimensionality Explosion in ISP Parameter Space

The number of parameters in a modern mobile-platform ISP far exceeds that of early embedded image-processing systems. For a typical flagship camera, the complete ISP parameter system may span the following dimensions:

**Module dimension**: A typical ISP pipeline contains 15–30 configurable modules, each with 5–50 parameters, yielding a total parameter count of 500–2,000.

**Illuminant/color-temperature dimension**: To accommodate different lighting conditions, modules such as AWB (Auto White Balance), CCM (Color Correction Matrix), and tone curves are typically configured at 3–7 standard color-temperature points (e.g., A / TL84 / D50 / D65 / D75 / H / HORIZON), with continuous runtime parameters generated by color-temperature interpolation.

**ISO/gain dimension**: Parameters for NR (Noise Reduction), EE (Edge Enhancement), and color saturation vary with sensor gain. They are typically set at 6–10 ISO levels, forming a strength-vs-gain curve.

**Scene/mode dimension**: Camera app shooting modes (Auto, Portrait, Night, Sport, Pro) correspond to different ISP parameter combinations; the number of modes is typically 5–15.

**Sensor dimension**: A single handset may carry three or more cameras (main, ultra-wide, telephoto), each using a different sensor and requiring an independent parameter set.

Expressed as an order-of-magnitude formula, the total parameter count is:

```
N_total ≈ N_modules × N_params_per_module × N_CCT × N_ISO × N_modes × N_sensors
         ≈ 20 × 15 × 5 × 8 × 8 × 3 ≈ 288,000 parameter entries
```

At this scale, manually managing parameter versions is practically infeasible; systematic engineering methods are essential.

### 1.2 The Necessity of Version Management: Parameter Regression Risk

**Parameter regression** refers to the phenomenon where optimizing one scene in a new tuning round inadvertently degrades image quality in other scenes. Without version management, parameter regression carries the following dangers:

- **Hard to detect**: Tuning engineers typically focus on the scene under active debug and do not proactively verify all historical scenes.
- **Hard to locate**: There is no record of when, what, or why a parameter was changed, making it difficult to trace which modification caused the regression.
- **Hard to recover**: Without historical snapshots, rollback relies on memory alone, making it extremely inefficient.

A typical parameter regression example: A tuning engineer raises the high-ISO segment of the NR strength curve to improve low-light ISO 3200 denoising, without noticing that the same parameter also affects backlit HDR scenes (where AE selects high ISO to avoid underexposure in shadows), causing backlit scenes to lose texture detail and ultimately leading to customer complaints. With complete version management and automated regression testing, this issue would have been caught automatically at the time of commit.

### 1.3 Core Goals of Parameter Management

An excellent ISP parameter management system must satisfy:

1. **Traceability**: Every parameter set maps to a unique version number, with queryable history of all modifications (which parameter changed, before/after values, reason, linked bug ID).
2. **Diffability**: Fast comparison of parameter differences between two versions, presented in a human-readable format.
3. **Rollback**: Fast restoration to any historical version without manual reconstruction.
4. **Auto-validation**: After a parameter change is submitted, objective metric tests run automatically to block changes that would cause regressions.
5. **Multi-device consistency**: Mass-production parameter behavior is predictable and quantifiable across devices from different batches.

## §2 Parameter Organization Architecture

### 2.1 Parameter Hierarchy

ISP parameters are typically organized in the following three-layer structure, progressing from global to specific:

**Layer 1: Global Parameters**
- Foundation parameters applicable to all scenes and modes
- Includes: sensor timing configuration, ISP pipeline module enable/disable switches, global gain offsets
- Change frequency: low (modified only during EVT phase or when sensor driver is updated)

**Layer 2: Scene Parameters**
- Parameter sets grouped by scene mode (daytime / night / portrait, etc.)
- Includes: AE strategy parameters per mode, NR-EE trade-off parameters, color preference parameters
- Change frequency: medium (frequently changed during DVT phase, stabilizes after PVT)

**Layer 3: Sensor-Specific Parameters**
- Calibration parameters bound to a specific sensor lot
- Includes: BLC values, LSC gain maps, AWB base gains, CCM matrices
- Change frequency: high (updated each time the sensor lot changes or recalibration occurs)

**Hierarchy rules**:
- Lower-layer parameters may override specific fields of upper-layer parameters.
- Merge order: Global parameters → Scene parameters (override) → Sensor parameters (override) → Runtime dynamic parameters

This hierarchical structure avoids the Cartesian-product explosion of "maintaining a full parameter set for every sensor × scene combination."

### 2.2 Choosing a Parameter Format

The choice of ISP parameter storage format affects toolchain development efficiency and mass-production reliability:

**XML format**:
- Pros: Clear structure, supports comments, strong readability, mature tooling (xpath, xmllint)
- Cons: Larger file size due to repeated tag overhead; XML parsers carry some RAM cost on embedded systems
- Suitable for: Tuning tool user interfaces, parameter documentation archiving

**JSON format**:
- Pros: More compact than XML, concise syntax, high-quality JSON libraries in nearly every language; natively supports arrays (suitable for LUT parameters)
- Cons: No comments in standard JSON; requires additional comment conventions (e.g., JSONC or a `_comment` field)
- Suitable for: Tuning framework parameter configuration, CI/CD automation toolchains

**Binary/bin format**:
- Pros: Smallest size (all structural markers removed), fastest read speed (suitable for direct `mmap` by embedded ISP drivers)
- Cons: Zero readability, diff tools cannot compare directly, requires companion binary conversion tools
- Suitable for: Mass-production factory parameters, the final form burned to device storage

**Engineering recommendation**: Use JSON/XML as the primary format for tuning and version control. At the mass-production release stage, automatically compile JSON/XML to the device-side bin format via automation scripts. Maintain a deterministic one-way compilation from JSON/XML (source format) → bin (target format), ensuring that bin file content can be fully reconstructed from the source format.

### 2.3 Common Patterns in Mainstream Platform Parameter Systems

ISP tuning frameworks across different SoC platforms differ in interface and tooling. Well-known examples include Qualcomm's **Chromatix XML** parameter system (used with the Spectra ISP and CamX HAL), MediaTek's **NDD (Noise Distribution Data)** noise characterization format (stores sensor noise model parameters indexed by gain/ISO), and HiSilicon's **IQ Tool** for Kirin ISP configuration. Despite these differences, they exhibit the following common patterns in parameter organization (the following are general ISP engineering principles and do not involve any platform-specific proprietary specifications):

> **Note on Qualcomm Chromatix:** Parameters are compiled from XML source into `.bin` binary files for on-device use, indexed by a multi-dimensional trigger vector `{sensor_mode, iso_gain, color_temperature, lux_index}`. At runtime, the 3A state machine (AE/AWB/AF) provides the current operating point, and the ISP driver performs interpolation to select the effective parameter values.

> **Note on MediaTek NDD:** NDD (Noise Distribution Data) is specifically a sensor noise characterization format that stores the noise model parameters (e.g., shot noise coefficient, read noise floor) at each gain/ISO operating point. It is distinct from the broader tuning parameter database.

**Common pattern 1: Trigger-indexed parameters**
- Parameters use ISO (gain) and color temperature (CCT, Correlated Color Temperature) as dual-axis trigger conditions, forming a 2D LUT.
- At runtime, the current parameters are obtained by bilinear interpolation from the ISO output of AE and the CCT output of AWB.

**Common pattern 2: Use-case separation**
- Parameter sets for "Still" and "Video" are maintained separately; they differ fundamentally in NR latency, EE strength, etc.
- Video parameters emphasize temporal stability (avoiding inter-frame parameter jumps), while still parameters pursue optimal single-frame quality.

**Common pattern 3: Separation of debug/tuning interface from production interface**
- During tuning, parameters are changed in real time via ADB (Android Debug Bridge) or a dedicated debug channel to verify effects.
- Once production parameters are finalized, the debug channel is typically closed to prevent exploitation.

### 2.4 Parameter Naming Conventions

Good parameter naming conventions are the foundation of version management. The recommended convention is:

```
<module_abbreviation>_<functional_description>_<dimension/unit>
```

Examples:
- `nr_luma_strength_iso800`: Luma noise-reduction strength at ISO 800
- `awb_r_gain_cct5000`: AWB R-channel gain at 5000 K color temperature
- `ee_gain_edge_high_iso200`: Sharpening gain for strong-edge regions at ISO 200
- `ltm_gain_limit_hdr_mode`: Local tone-mapping gain upper limit in HDR mode

Avoid using meaningless numeric sequences (e.g., `param_001`) or overly abbreviated names (e.g., `nrs`), which make diff tool output difficult to interpret.

## §3 Version Control Practices

### 3.1 Git-LFS for Parameter File Management

ISP parameter files have special characteristics: JSON/XML source files are typically 1–10 MB and can be managed with ordinary Git, while bin-format files may reach tens to hundreds of MB (containing large volumes of floating-point LUT data) and require **Git-LFS (Git Large File Storage)**.

**Git-LFS configuration example**:

```bash
# Initialize Git-LFS
git lfs install

# Track bin parameter files and LSC gain maps
git lfs track "*.bin"
git lfs track "*.lsc"
git lfs track "calibration_data/**"

# View .gitattributes configuration
cat .gitattributes
# Output:
# *.bin filter=lfs diff=lfs merge=lfs -text
# *.lsc filter=lfs diff=lfs merge=lfs -text

# Normal commit and push (LFS files are uploaded to the LFS server; only pointers are stored in the Git repo)
git add params/sensor_lsc_gains.bin
git commit -m "feat: update LSC gain map for sensor lot B"
git push origin main
```

**Recommended directory structure**:

```
isp_params/
├── global/
│   ├── pipeline_config.json       # Global pipeline enable/disable configuration
│   └── sensor_timing.json         # Sensor timing parameters
├── scenes/
│   ├── auto_mode/
│   │   ├── ae_strategy.json       # AE strategy parameters
│   │   ├── nr_ee_curve.json       # NR-EE strength curves
│   │   └── color_preference.json  # Color preference (saturation/hue)
│   ├── night_mode/
│   │   └── ...
│   └── portrait_mode/
│       └── ...
├── calibration/                   # Sensor calibration data (Git-LFS)
│   ├── main_cam/
│   │   ├── blc_v1.2.json          # Black level calibration
│   │   ├── lsc_d65_v2.1.bin       # LSC gain map under D65
│   │   ├── awb_gains_v3.0.json    # AWB color-temperature gain curve
│   │   └── ccm_multi_v2.3.json    # Multi-CCT CCM matrices
│   └── tele_cam/

├── releases/                      # Mass-production release snapshots

└── CHANGELOG.md                   # Parameter change log
```

### 3.2 Parameter Diff Tools

Standard `git diff` works on JSON/XML parameter files but is not intuitive enough — changes in array-type LUT parameters (e.g., a 50-control-point tone curve) appear as large-block replacements, making it difficult to quickly identify key differences.

In engineering practice, a **semantic parameter diff tool** is typically developed to output structured parameter-difference reports organized by module. Example output format:

```
=== ISP Parameter Diff Report ===
Comparing: v1.2.0 → v1.2.1
Changed modules: NR (2 parameters), EE (1 parameter)

[NR] nr_luma_strength_iso800:   0.65 → 0.72  (+10.8%)
[NR] nr_luma_strength_iso1600:  0.78 → 0.83  (+6.4%)
[EE] ee_gain_edge_high_iso200:  1.35 → 1.30  (-3.7%)

Risk assessment: MEDIUM (NR strength raised; recommend verifying ISO 800-1600 MTF50)
```

This output format allows tuning engineers to quickly assess the impact scope of a change during code review.

### 3.3 Automated Regression Testing Pipeline

A **CI/CD (Continuous Integration / Continuous Delivery)** pipeline is the core safeguard against parameter regression. A typical pipeline includes the following stages:

**Stage 1: Schema Validation**
- Verifies that submitted parameter files conform to the JSON Schema specification
- Checks parameter value ranges (e.g., gain parameters do not exceed [0, 4]; NR strength is within [0, 1])
- Duration: < 1 minute

**Stage 2: Static Parameter Analysis**
- Checks constraint relationships between parameters (e.g., `ee_gain_max` must be ≥ `ee_gain_min`)
- Warns about anomalous values (e.g., NR strength jumps > 20% between adjacent ISO points)
- Duration: < 2 minutes

**Stage 3: Offline Rendering Validation**
- Applies parameters to a pre-stored standard RAW image dataset (typically 50–200 images covering representative scenes)
- Computes objective metrics (MTF50, SNR, ΔE2000) on output images
- Compares with the baseline version; blocks the commit if key metrics drop below a threshold
- Duration: 10–60 minutes (depending on dataset size and rendering speed)

**Stage 4: Multi-Scene Coverage Report**
- Generates a list of scenes affected by the current change
- Visualizes the impact of changed parameters across all ISO × CCT × mode combinations
- Duration: < 5 minutes

**Pipeline trigger conditions**:
- Stages 1–2 triggered automatically when a developer submits an MR (Merge Request)
- Stages 3–4 triggered automatically when an MR is approved and ready to merge into the main branch
- Full nightly regression runs the complete pipeline every night

### 3.4 Risk Assessment Matrix for Parameter Changes

Not all parameter changes carry the same risk. The following risk assessment matrix helps tuning engineers decide the required validation effort for a change:

| Change Scope | Change Magnitude | Risk Level | Required Validation |
|---|---|---|---|
| NR strength at a single ISO point | < 10% | Low | MTF50/SNR at that ISO point |
| NR strength curve raised across the board | 5%–20% | Medium | Full-ISO-range MTF50/SNR + subjective sampling |
| CCM matrix update | Any | Medium-High | Full-CCT ΔE2000 + AWB accuracy |
| LSC gain map update | Any | Medium | Full-scene uniformity + AWB sampling validation |
| AE strategy change | Any | High | Full-scene video frame-by-frame luminance stability |
| Tone mapping curve change | Any | High | Full-scene DR/highlight protection + subjective MOS |

## §4 Multi-Scene Switching Logic

### 4.1 Automatic Illuminant Detection and Parameter Interpolation

At runtime, the ISP interpolates the current parameter values from a multi-CCT parameter table based on the current color temperature estimated by the AWB algorithm.

**Dual-CCT interpolation** is the most commonly used approach:

```
param_current = param_cct_low × w_low + param_cct_high × w_high

w_low  = (cct_high - cct_current) / (cct_high - cct_low)
w_high = (cct_current - cct_low) / (cct_high - cct_low)
```

where `cct_low` and `cct_high` are the two calibrated color-temperature points closest to the current CCT.

**Multi-dimensional interpolation** is applied in joint AE-AWB tuning scenarios:
- For parameters that depend on both ISO and CCT (e.g., NR strength behavior at high CCT and high ISO), bilinear interpolation is used on the 2D ISO×CCT grid.
- **ISO-domain interpolation is typically performed in log scale** (i.e., interpolate in log₂(ISO) space rather than linear ISO), because human perception of noise is approximately logarithmic with gain and the noise variance grows proportionally with gain. Performing linear interpolation in raw ISO space would over-weight the high-ISO endpoint and produce a non-smooth perceptual transition.
- Care must be taken regarding the physical meaning of parameters during interpolation — for example, CCM matrix interpolation should linearly interpolate matrix elements, not interpolate ΔE errors.

**Interpolation smoothness guarantee**: When the AWB-estimated CCT changes rapidly (e.g., moving from indoors to outdoors), the interpolated parameter update must not produce a step change. A low-pass filter (time constant approximately 5–15 frames) should be applied to the CCT estimate.

### 4.2 Scene-Recognition-Driven Dynamic Parameter Scheduling

Modern camera apps typically include a scene detection function that automatically identifies the type of content being captured (portrait, landscape, food, night scene, etc.) and switches to the correspondingly optimized ISP parameter set.

**Rule-based scene recognition**:
- Luminance histogram statistics + CCT estimate → night/day judgment
- AE statistics scene contrast ratio → backlight judgment
- AF (Auto Focus) distance + face detection → portrait mode trigger

**AI-based scene recognition** (mainstream approach in recent years):
- A lightweight scene-classification model (e.g., quantized MobileNetV2) runs on the NPU (Neural Processing Unit)
- Outputs scene categories and confidence scores (multi-label, e.g., "portrait: 0.85, indoor: 0.72")
- Dynamically loads the corresponding parameter set based on scene classification results; weighted fusion of multiple scene parameters with confidence > 0.5

**Parameter-switching hysteresis**:
- Scene recognition results may fluctuate between adjacent frames (e.g., border-case "indoor/outdoor" switching)
- Parameter switching requires hysteresis: entering a new scene requires N consecutive frames (e.g., 10–20 frames) of consistent recognition before executing the switch
- The switching process uses a smooth transition (5–10 frames to complete gradually), avoiding visual jumps

### 4.3 Time/Location-Triggered Parameter Switching

Some high-end camera apps leverage the device's time and geographic location to assist parameter decisions:

**Time-triggered**:
- Sunrise/sunset period (solar elevation angle < 10°, computed from geographic location + time) → automatic switch to golden-hour color parameters (warm tone enhancement, higher highlight protection)
- Night (solar elevation angle < −6°, end of civil twilight) → night mode parameters pre-loaded

**Location-triggered** (high-end scenarios, not universal):
- Indoor/outdoor determination (combined assessment of Wi-Fi signal strength, barometer, and light intensity)
- High-altitude scenario (atmospheric pressure < 80% of standard) → stronger UV, may adjust exposure strategy

The parameter management complexity for such features is higher, requiring the addition of **trigger condition** fields to the parameter system and the design of priority rules (when multiple trigger conditions are satisfied simultaneously, which parameter set takes precedence).

## §5 Mass-Production Parameter Validation and Release

### 5.1 Parameter Consistency Check

Before mass production, the following consistency checks must be performed on the parameter files themselves (independent of hardware):

**Schema completeness**:
- Verify that all required fields exist
- Check parameter range validity
- Verify that LUT array lengths match format declarations

**Semantic consistency**:
- AWB gain curve monotonicity check: R_gain should monotonically decrease as CCT increases; B_gain should monotonically increase
- NR strength curve continuity check: the strength difference between adjacent ISO points must not exceed 30% (to prevent visible quality jumps at ISO transitions)
- Tone mapping curve monotonicity check: output values must be strictly monotonically increasing

**Cross-module consistency**:
- LSC gain map resolution matches the LSC grid size supported by the ISP hardware
- Each row of the CCM matrix sums to a value within [0.98, 1.02] (approximately preserving luminance normalization)
- The color-temperature range covered by AWB calibration encompasses the CCT index range of NR/CCM parameters

### 5.2 Multi-Device Offline Validation

Before a mass-production release, offline validation must be executed on a batch of devices (typically 20–50 units) to confirm performance under different individual variation:

**Validation items**:

| Validation Item | Evaluation Method | Pass Criterion |
|---|---|---|
| MTF50 consistency | Slant-edge method, 9-grid positions | σ(MTF50) < 50 lp/ph |
| AWB color-temperature accuracy | Standard illuminant gray-card test | CCT deviation < ±200 K |
| ΔE2000 distribution | ColorChecker under D65 | Mean < 4.5, σ < 0.8 |
| LSC uniformity | Uniform light-box capture | Max corner luminance deviation < 5% |
| Dark-field noise | Lens-covered capture, mean/std per channel | Per-channel std dev < 3 DN |

**Statistical analysis**:
- Compute mean, standard deviation, max, and min for each validation item
- Compute Cpk (Process Capability Index); critical items require Cpk > 1.33
- If an outlier is found (sample deviating from the mean by > 3σ), investigate that device's module assembly or sensor lot

### 5.3 Factory Parameter Programming Workflow

Mass-production parameter programming is the final deployment step and requires strict process control:

**Programming format**:
- Source parameters (JSON/XML) are converted to device-side bin format by a compilation tool
- The bin file includes a version header (magic number + version number + CRC32 checksum)
- At device startup, the ISP driver verifies the CRC32; on failure, it falls back to factory default parameters

**Programming verification**:
- After programming, the device automatically executes a Built-in Self-Test (BIST)
- BIST captures standard test images (reference patterns stored in ROM before factory shipping) to verify that key parameters have taken effect
- BIST pass rate must reach 100% before the device is allowed to ship

**Version recording**:
- The device's parameter version number is written to persistent storage (e.g., nvram or misc partition) and supports remote query
- During OTA parameter updates, both old and new version numbers are recorded, supporting A/B parameter rollback

### 5.4 Field Update Mechanism

After mass production, updating ISP parameters via OTA (Over-The-Air) is an important means of continuously optimizing image quality.

**OTA parameter update architecture**:
```
Tuning engineer → Parameter Git repo → CI automated validation → Parameter packaging server
    → Push to device OTA channel → Device-side parameter update service → ISP driver reloads
```

**Security considerations**:
- Parameter packages use digital signatures (e.g., RSA-2048); the device verifies the signature before executing the update
- Parameter updates cannot modify firmware; only the parameter storage area can be modified (enforced via access control)
- The current parameter version is automatically backed up before an update, supporting manual rollback by the user (via a "Restore factory camera settings" option)

**Gradual rollout**:
- A new parameter version is first pushed to internal test users (1%–5% of devices)
- Automated quality metrics are collected (e.g., capture success rate, crash rate, user satisfaction score)
- If no anomalies are found, the rollout percentage is gradually expanded (10% → 50% → 100%)

**A/B testing in production**:
- To validate competing parameter variants (e.g., two NR-EE trade-off tunings), devices or users are randomly assigned to Group A (existing parameters) or Group B (candidate parameters)
- Both groups operate in production conditions simultaneously, eliminating time-of-day and scene-distribution confounders
- Quality metrics (user-reported satisfaction scores, technical IQA signals derived from uploaded images) are collected and compared statistically
- A new parameter set is promoted to full rollout only when Group B demonstrates a statistically significant improvement (or at minimum no regression) over Group A

## §6 Code Examples

### 6.1 Parameter Version Diff Tool

```python
"""
isp_param_diff.py
ISP parameter semantic difference comparison tool
Usage: python isp_param_diff.py params_v1.json params_v2.json
"""

import json
import sys
import argparse
import math
from pathlib import Path

# Risk mapping: changes to certain modules require more attention
RISK_WEIGHTS = {
    'ccm':    'HIGH',
    'awb':    'HIGH',
    'ae':     'HIGH',
    'lsc':    'MEDIUM',
    'nr':     'MEDIUM',
    'ee':     'LOW',
    'gamma':  'MEDIUM',
    'csc':    'LOW',
}

def flatten_dict(d, prefix='', sep='.'):
    """Flatten nested JSON to key.subkey.field format"""
    items = {}
    for k, v in d.items():
        new_key = f"{prefix}{sep}{k}" if prefix else k
        if isinstance(v, dict):
            items.update(flatten_dict(v, new_key, sep))
        elif isinstance(v, list) and all(isinstance(x, (int, float)) for x in v):
            # Array type (LUT): store as tuple for comparison
            items[new_key] = tuple(v)
        else:
            items[new_key] = v
    return items

def compare_lut(lut_old, lut_new, key):
    """Compare two numeric arrays (LUTs), return summary difference"""
    if len(lut_old) != len(lut_new):
        return f"  [{key}] Length change: {len(lut_old)} → {len(lut_new)}"
    diffs = [abs(n - o) for o, n in zip(lut_old, lut_new)]
    max_diff = max(diffs)
    max_idx  = diffs.index(max_diff)
    changed  = sum(1 for d in diffs if d > 1e-6)
    if changed == 0:
        return None
    pct_change = (lut_new[max_idx] - lut_old[max_idx]) / (abs(lut_old[max_idx]) + 1e-9) * 100
    return (f"  [{key}] LUT change: {changed}/{len(lut_old)} points changed, "
            f"max diff {max_diff:.4f} (index {max_idx}, {pct_change:+.1f}%)")

def compute_risk(changed_keys):
    """Estimate risk level based on changed parameter keys"""
    risk_levels = {'HIGH': 0, 'MEDIUM': 0, 'LOW': 0}
    for key in changed_keys:
        module = key.split('.')[0].lower()
        level = RISK_WEIGHTS.get(module, 'LOW')
        risk_levels[level] += 1
    if risk_levels['HIGH'] > 0:
        return 'HIGH', risk_levels
    elif risk_levels['MEDIUM'] > 0:
        return 'MEDIUM', risk_levels
    return 'LOW', risk_levels

def diff_isp_params(file_old, file_new, verbose=True):
    """
    Compare two ISP parameter JSON files and output a semantic diff report.
    Returns a list of diff entries.
    """
    with open(file_old, 'r', encoding='utf-8') as f:
        params_old = json.load(f)
    with open(file_new, 'r', encoding='utf-8') as f:
        params_new = json.load(f)

    flat_old = flatten_dict(params_old)
    flat_new = flatten_dict(params_new)

    all_keys = set(flat_old.keys()) | set(flat_new.keys())
    added    = [k for k in all_keys if k not in flat_old]
    removed  = [k for k in all_keys if k not in flat_new]
    changed  = []
    diff_lines = []

    for key in sorted(all_keys):
        if key in added or key in removed:
            continue
        v_old = flat_old[key]
        v_new = flat_new[key]
        if isinstance(v_old, tuple) and isinstance(v_new, tuple):
            # LUT comparison
            lut_diff = compare_lut(v_old, v_new, key)
            if lut_diff:
                changed.append(key)
                diff_lines.append(lut_diff)
        elif v_old != v_new:
            changed.append(key)
            if isinstance(v_old, float) and isinstance(v_new, float):
                pct = (v_new - v_old) / (abs(v_old) + 1e-9) * 100
                diff_lines.append(f"  [{key}]  {v_old:.4f} → {v_new:.4f}  ({pct:+.1f}%)")
            else:
                diff_lines.append(f"  [{key}]  {v_old!r} → {v_new!r}")

    if verbose:
        print(f"\n{'='*60}")
        print(f"ISP Parameter Diff Report")
        print(f"  Old version: {Path(file_old).name}")
        print(f"  New version: {Path(file_new).name}")
        print(f"{'='*60}")
        print(f"Added: {len(added)}, Removed: {len(removed)}, Changed: {len(changed)}")

        if added:
            print(f"\n[Added parameters ({len(added)})]")
            for k in sorted(added):
                print(f"  +[{k}] = {flat_new[k]!r}")

        if removed:
            print(f"\n[Removed parameters ({len(removed)})]")
            for k in sorted(removed):
                print(f"  -[{k}] = {flat_old[k]!r}")

        if changed:
            print(f"\n[Changed parameters ({len(changed)})]")
            for line in diff_lines:
                print(line)

        # Risk assessment
        risk_level, risk_counts = compute_risk(changed + added + removed)
        print(f"\n{'='*60}")
        print(f"Risk assessment: {risk_level}")
        print(f"  HIGH-risk module changes: {risk_counts['HIGH']} items")
        print(f"  MEDIUM-risk module changes: {risk_counts['MEDIUM']} items")
        print(f"  LOW-risk module changes: {risk_counts['LOW']} items")
        if risk_level == 'HIGH':
            print("  >>> Recommendation: Run full-scene objective metric regression + subjective evaluation")
        elif risk_level == 'MEDIUM':
            print("  >>> Recommendation: Run objective metric tests for affected modules")
        else:
            print("  >>> Recommendation: Quick verification of changed scenes; full regression not required")
        print(f"{'='*60}\n")

    return {'added': added, 'removed': removed, 'changed': changed,
            'risk': risk_level}
```

### 6.2 Multi-Device Parameter Consistency Evaluation Script

```python
"""
multi_device_consistency.py
Batch validation of ISP parameter consistency across multiple devices
(Uses objective metric result files; no direct device connection required)
"""

import numpy as np
import json
from pathlib import Path
import matplotlib.pyplot as plt

def compute_cpk(values, usl, lsl):
    """
    Compute the Process Capability Index (Cpk).
    values: list of sample values
    usl: Upper Specification Limit
    lsl: Lower Specification Limit
    """
    mu    = np.mean(values)
    sigma = np.std(values, ddof=1)  # unbiased standard deviation
    if sigma < 1e-9:
        return float('inf')  # zero variance, infinite process capability
    cpu = (usl - mu) / (3.0 * sigma)
    cpl = (mu - lsl) / (3.0 * sigma)
    return min(cpu, cpl)

def analyze_multi_device(metrics_list, spec_limits, device_ids=None):
    """
    Analyze objective metric consistency across multiple devices.

    Args:
        metrics_list: list of dict, objective metric dictionary per device
            Example: [{'mtf50_center': 1750, 'delta_e_mean': 3.2, 'snr_1000lux': 43.1}, ...]
        spec_limits: dict, specification limits for each metric
            Example: {'mtf50_center': (1400, 2200), 'delta_e_mean': (0, 5.0), ...}
        device_ids: list of str, device serial number list
    Returns:
        analysis_report: dict, statistical results for each metric
    """
    if device_ids is None:
        device_ids = [f"DEV_{i+1:03d}" for i in range(len(metrics_list))]

    n_devices = len(metrics_list)
    all_metrics = list(spec_limits.keys())
    report = {}

    print(f"\n{'='*65}")
    print(f"Multi-Device Consistency Analysis Report  (N={n_devices} devices)")
    print(f"{'='*65}")
    print(f"{'Metric':<25} {'Mean':>8} {'Std':>8} {'Min':>8} {'Max':>8} {'Cpk':>7} {'Status':>6}")
    print(f"{'-'*65}")

    for metric in all_metrics:
        values = np.array([m[metric] for m in metrics_list])
        lsl, usl = spec_limits[metric]
        cpk = compute_cpk(values, usl, lsl)
        status = "PASS" if cpk >= 1.33 else ("WARN" if cpk >= 1.0 else "FAIL")

        report[metric] = {
            'values':  values.tolist(),
            'mean':    float(np.mean(values)),
            'std':     float(np.std(values, ddof=1)),
            'min':     float(np.min(values)),
            'max':     float(np.max(values)),
            'cpk':     cpk,
            'status':  status,
            'lsl':     lsl,
            'usl':     usl,
        }

        print(f"{metric:<25} {np.mean(values):>8.2f} {np.std(values,ddof=1):>8.3f} "
              f"{np.min(values):>8.2f} {np.max(values):>8.2f} {cpk:>7.2f} {status:>6}")

    print(f"{'='*65}")

    # Identify outlier devices (any metric out of spec)
    outlier_devices = set()
    for metric, data in report.items():
        lsl, usl = data['lsl'], data['usl']
        for i, v in enumerate(data['values']):
            if v < lsl or v > usl:
                outlier_devices.add(device_ids[i])

    if outlier_devices:
        print(f"\nOutlier devices (at least one metric out of spec): {sorted(outlier_devices)}")
        print("Recommend investigating module assembly and sensor lot for these devices.")
    else:
        print(f"\nAll {n_devices} devices pass all metrics within specification.")

    return report

def plot_consistency(report, save_path=None):
    """Visualize multi-device consistency results (box plot + spec lines)"""
    metrics = list(report.keys())
    n = len(metrics)
    fig, axes = plt.subplots(1, n, figsize=(4 * n, 5))
    if n == 1:
        axes = [axes]

    for ax, metric in zip(axes, metrics):
        data   = report[metric]
        values = data['values']
        bp = ax.boxplot(values, patch_artist=True,
                        boxprops=dict(facecolor='lightblue', color='blue'),
                        medianprops=dict(color='red', linewidth=2))
        ax.axhline(data['usl'], color='green', linestyle='--', linewidth=1.5,
                   label=f"USL={data['usl']:.1f}")
        ax.axhline(data['lsl'], color='orange', linestyle='--', linewidth=1.5,
                   label=f"LSL={data['lsl']:.1f}")
        ax.set_title(f"{metric}\nCpk={data['cpk']:.2f} ({data['status']})",
                     fontsize=9)
        ax.legend(fontsize=7)
        ax.grid(True, alpha=0.3)

    plt.suptitle("Multi-Device ISP Metric Consistency Analysis", fontsize=12, fontweight='bold')
    plt.tight_layout()
    if save_path:
        plt.savefig(save_path, dpi=150, bbox_inches='tight')
    plt.show()
```
